Configuration

The generated config is a starting point. The important part is matching Hub inventory and real local paths.

Identity must match inventory

The Hub resolves incoming events by organization, project, environment, app, service, and agent id. If the dashboard app slug is demo-app, the agent site app must also be demo-app. A mismatch is rejected instead of silently creating unknown inventory.

identity:
  org: demo-org
  project: demo-site
  environment: production
  host: demo-prod
  agent_id: agt_demo_prod

sites:
  - slug: demo-site
    kind: wordpress
    app: demo-app
    service: frontend
    root: /mnt/sites/demo-site

Collectors

Enable only the collectors that make sense for that node. Pantheon-style hosted sites may disable files or config monitoring if the agent cannot read those paths. The dashboard shows disabled and ignored areas so an operator can see what the node is not watching.

files:
  enabled: true
  profiles: [wordpress]
  exclude:
    - /mnt/sites/demo-site/wp-content/cache
    - /mnt/sites/demo-site/wp-content/uploads/backups
coverage:
  enabled: true
browser_crawl:
  enabled: true
  rendered: false

Databases

Use environment variables for DSNs. Profiles decide which tables and fields matter: WordPress users and high-signal options, PrestaShop employees and modules, Mautic users, Laravel auth tables, and Yii2 RBAC assignments. Static, React, and Node.js profiles do not collect database snapshots by default.

databases:
  - name: demo-db
    engine: mysql
    dsn_env: AEGRAIL_DB_DEMO_DSN
    profile: wordpress
    table_prefix: mrk_
    schedule: 5m

Secrets

Do not store DSNs, passwords, API keys, cookies, or tokens in the YAML. Use environment variables or your container secret mechanism. The agent redacts sensitive values before queueing evidence, but the cleanest secret is the one that never enters the config file.
