Agents and collectors

An agent is the local observer for one node. Collectors decide what is worth sending to the Hub.

Agent model

Run one agent per site or instance when you want clear ownership. A node can watch a single application root, database profile, log set, and public URL. The Hub shows the node, its collector status, ignored paths, and last scan times.

Files collector

The file collector walks configured roots, tracks size, hash, ctime, mtime, and path metadata, and skips known-noisy areas such as caches, generated uploads, backups, and the agent queue. Large files can be tracked by metadata without hashing. Directory state is cached so unchanged trees are cheaper to revisit.

Grouping matters: a new plugin or module should become one useful finding with related files, not hundreds of separate warnings.

Database collector

Database profiles watch high-signal entities. WordPress checks users, roles, active plugins, site URL, home URL, admin email, registration settings, and default role. PrestaShop focuses on employees and modules. Mautic, Laravel, and Yii2 RBAC profiles focus on user and permission tables that matter for those apps.

Logs collector

Log scanning is tuned for security and operational signals: admin logins, password reset requests, suspicious paths, known bad patterns, and Tor IP checks. Static assets, email redirect noise, and routine platform traffic should be filtered before they become findings.

Browser and config collectors

The browser collector fetches configured URLs and records client-visible scripts, headers, icons, links, and important page metadata. Rendered crawling is optional and only runs when a working Chrome-compatible browser is available. Config coverage checks whether expected config files and Nginx/Apache site definitions are visible to the agent.
